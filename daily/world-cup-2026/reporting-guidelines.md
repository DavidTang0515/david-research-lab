# 2026 世界杯日报规范

本规范适用于 2026 年世界杯赛事期间的每日动态分析，是 `daily/world-cup-2026/` 下世界杯日报、动态总览和自动化执行的正式规则文件。

本栏目不是单纯赛果搬运，也不是纯赛后专栏。默认目标是：**动态日报 + 动态总览 + 赛后专栏式判断力**。写法以 `2026-06-24.md` 的完整框架为主要参考：先用一个清晰比赛日主线统领全文，再按赛果、实际比分与 xG、精彩瞬间、格局影响、今日看点、争冠梯队、热门重估、前三名预测、亮眼球员、战术观察、结论、来源和数据覆盖说明展开。

## 1. 发布条件

按北京时间 / Asia/Singapore 口径判断是否发布。

满足以下条件才可发布日报：

1. 当前北京时间比赛窗口内所有应纳入日报的比赛均已结束。
2. 这些比赛已得到固定来源确认；若 FIFA / ESPN 当次运行中不可稳定回读，但 Reuters / AP / BBC / The Guardian 等权威媒体已明确确认终场赛果，可用于当期发布，并标注“待 FIFA / ESPN 回读确认”。
3. 本比赛窗口尚未生成过 David Research Lab 世界杯动态日报。

若未满足条件，不要发布；只说明已确认赛果、缺失数据、检查来源、完成部分、GitHub 连接器 / 写权限状态、commit 状态和同步状态。

## 2. 数据时点与比赛日口径

- 每篇日报必须写明生成时间、数据截止时间、官方比赛日和北京时间实际发生时段。
- 日期以北京时间 / Asia/Singapore 为主要展示口径，同时说明赛事当地日期，避免跨时区混淆。
- 只把已经结束并得到可靠来源确认的比赛写入“昨日及今晨赛果 / 比赛结果速览”。正在进行或尚未开始的比赛只能放入“今日赛程与关键看点 / 下一比赛日看点”。
- 如果用户要求“昨天到今天版”，覆盖窗口应明确写成“北京时间 YYYY-MM-DD 00:00 至本次生成时间”，并只整理该窗口内已经结束并确认的比赛。
- 如果同日多次更新，要说明是“增量回读”还是“完整窗口重写”，避免读者误以为新增了比赛。

### 2.1 北京时间窗口强制校验

生成日报前必须先完成以下三步：

1. 读取 FIFA / ESPN 赛程；若固定来源不可稳定回读，可读取可靠赛程源作临时核查。
2. 将每场比赛的官方当地开球时间逐场换算为北京时间 / Asia/Singapore（UTC+8）。
3. 用换算后的北京时间日期判断：
   - 哪些比赛属于本期赛果窗口；
   - 哪些比赛属于“今日赛程 / 下一比赛日看点”；
   - 哪些比赛只能放入“后续赛程提示”。

不得把北美当地日期直接当成北京时间日期。不得因为赛事当地同一天就把比赛提前写进北京时间下一比赛日。

日报开头建议写成：

```markdown
官方比赛日：北美当地 YYYY-MM-DD。  
北京时间实际发生时段：YYYY-MM-DD HH:mm 至 YYYY-MM-DD HH:mm。  
赛程核查：以下赛程均已按 Asia/Singapore / UTC+8 换算；不属于该北京时间窗口的比赛不列入本期赛程。
```

### 2.2 下一比赛日栏目写法

“今日赛程与关键看点 / 下一比赛日看点”只允许列出**紧接着本期之后的北京时间比赛窗口**。

如果要提示更远赛程，必须另设“后续赛程提示”，并明确日期，不得与下一比赛日混写。

示例：

- 北京时间 2026-07-06 日报之后，下一比赛日是北京时间 2026-07-07：葡萄牙 vs 西班牙、美国 vs 比利时。
- 阿根廷 vs 埃及、瑞士 vs 哥伦比亚属于北京时间 2026-07-08，不得写入 2026-07-07 下一比赛日。

## 3. 来源优先级

所有日报必须先读取并遵守 [`source-policy.md`](./source-policy.md)。该文件指定每类数据的主来源、固定备用来源和缺失处理方式。

固定口径如下：

- 赛程、比分、比赛状态、小组积分、晋级、淘汰、首发、换人、进球和红黄牌：FIFA 官方为主，ESPN 为固定备用。
- 基础技术统计：优先 FIFA；FIFA 缺失时整场使用 FotMob，不拼接两家数据。
- xG：FBref / StatsBomb 为主；FBref / StatsBomb 暂未更新或不可稳定取得时，整场使用 FotMob xG 备用；同一场比赛不得混用两种 xG 口径。
- 重大机会和球员评分：仅使用 FotMob；FBref 不替代这两项，缺失时不写。
- 阵容、伤停和停赛：国家队官方为主，FIFA、Reuters、AP 为备用。
- 比赛过程和战术背景：Reuters 为主，AP、BBC、The Guardian、ESPN 为备用。
- 争冠概率模型：Opta Analyst 为主；仅在其未更新且 Goldman Sachs 有明确公开更新时补充后者。
- 默认不使用赔率。

### 3.1 数据使用原则

- 每场重点比赛都应主动检查可获得的技术统计、首发与换人、关键球员、伤停和纪律信息。
- 优先核对射门、射正、预期进球（xG）、重大机会、门将扑救、红牌和关键换人。
- xG 等模型数据必须注明数据机构；不同机构口径不得直接混用。
- 控球率、传球成功率、角球和跑动距离只在能够解释战术或比赛走势时使用。
- 数据用于检验和约束判断，不为了显得完整而堆砌。
- 正文只展示对赛果、晋级形势、球队评价或前三名预测有解释力的数据。
- 无法取得可靠数据时，应明确说明，不用媒体形容词代替统计事实。
- 每篇日报必须在文末增加“数据覆盖说明”，交代赛果、技术统计、xG、重大机会、球员评分、伤停、Reuters 检查和争冠模型的实际来源与缺失情况。

## 4. xG 获取、补录与展示

xG 是比赛分析中的重要参考数据，但不得贯穿全文、不得把日报主线写成 xG 专题。

每次生成或更新日报时，必须主动检查每场已结束重点比赛的 xG：

1. 先查 FBref / StatsBomb。
2. 若 FBref / StatsBomb 暂未更新或不可稳定取得，再查 FotMob xG。
3. 若 FotMob 页面能稳定回读 Top stats 或比赛页 xG，则使用 `FotMob xG`；若页面说明技术统计由 Opta 提供，可在说明中写作 `FotMob / Opta xG`。
4. 同一场比赛不得混用 FBref / StatsBomb 与 FotMob xG。
5. 不要让 xG 表长期为空；只有 FBref / StatsBomb 和 FotMob 都不可稳定取得时，才写“xG 暂无可靠数据”。
6. 如果日报已发布但后来在同一自动化运行或后续回读中稳定取得 xG，允许对既有日报做“xG 补录”，不得新建重复日报。
7. xG 补录时必须写明补录时间、来源和是否混用。
8. 重大机会和球员评分仍仅使用 FotMob，FBref 不替代这两项。

xG 只放在“昨日及今晨赛果 / 比赛结果速览”或“比赛分析”附近的简洁表格中。推荐表头：

| 比赛 | 实际比分 | xG 来源与数值 | 简要解读 |
|---|---:|---:|---|
| 巴西 vs 挪威 | 巴西 1-2 挪威 | FotMob xG：巴西 2.61 - 挪威 1.05（FBref / StatsBomb 暂未稳定取得） | 巴西机会质量占优但输球，说明点球、门将和终结效率改变了结果。 |

解读只需说明比分是否基本符合机会质量、是否存在低 xG 高效率、是否存在 xG 占优但未赢球、是否被门将 / 终结 / 点球稀释等。不要在争冠梯队、热门重估、球员评价、战术观察等章节反复引用 xG，除非实际比分与 xG 明显背离且对判断非常关键。

## 5. 中文球队名与中文球员名

日报面向中文阅读者，正文应以中文名为主。

- 球队名尽量使用中文：法国、阿根廷、葡萄牙、西班牙、英格兰、巴西、挪威、摩洛哥、美国、比利时等。
- 球员名尽量使用中文译名；首次出现或小节标题出现时，在后面括号加原名，例如：
  - 哈兰德（Erling Haaland）
  - 贝林厄姆（Jude Bellingham）
  - 哈里·凯恩（Harry Kane）
  - 姆巴佩（Kylian Mbappé）
  - 梅西（Lionel Messi）
  - C 罗（Cristiano Ronaldo）
- xG 表、赛果表、前三预测表也应使用中文球队名。
- 来源标题、URL 和媒体原文标题可以保留英文，避免破坏引用。
- 不确定中文译名时，优先采用常见中文译名；无法确认时保留原名，但不得在同一篇中中英文混乱切换。

## 开头总述

用 1-3 段写清楚本比赛日主线。不要只罗列比分，而要说明：谁升温、谁降温、谁回到主动区、谁出现结构性隐患、谁影响淘汰赛路径。

## 7. 标题和开头写法

标题应尽量包含“比赛日主线”，不要只写日期。例如：

- `世界杯动态日报｜6月23日比赛日：葡萄牙大胜回血，英格兰被拖入泥潭`
- `世界杯动态日报｜昨日到今日版：阿根廷稳住路径，葡萄牙进入硬线`

开头总述必须回答三个问题：

1. 这个比赛日最重要的主线是什么？
2. 哪些球队的评价发生变化？
3. 这种变化是积分变化、路径变化、伤停变化，还是比赛内容变化？

## 8. 赛果分析

每场重点比赛至少说明：

- 最终比分。
- 决定比赛走向的关键因素。
- 能够支持上述判断的关键比赛数据。
- 关键球员、重要换人、伤停或停赛影响。
- 对本组积分、出线概率、最佳第三名和淘汰赛落位的影响。
- 哪支球队的评价应上调或下调。

非重点比赛可只写比分、关键球员和晋级影响；重点比赛再增加数据对照。具体展示哪些指标，由其解释价值决定。

不得把控球率、射门数等单项数据直接等同于球队强弱。强队大胜要判断对手强度是否足够；0-0 或小胜要区分是结构性问题、效率问题，还是小组赛管理。

## 9. 最精彩的瞬间

这一节必须选择真正影响比赛叙事、球队处境或赛事格局的瞬间，不只是罗列进球。

可选类型包括：

- 关键进球或补时绝杀。
- 老将纪录或核心复出。
- 门将关键扑救。
- 伤退、红牌或关键换人。
- 最后阶段横梁、门线解围或点球。
- 让球队从风险区回到主动区的节点。

每个瞬间必须写“为什么重要”。

---

## 10. 对赛事走向的影响

在“小组和晋级格局影响”基础上，进一步说明对以下内容的影响：

- 小组出线和头名竞争。
- 最佳第三名比较。
- 32 强或后续淘汰赛路径。
- 强队评价和夺冠格局。
- 球队心理状态和轮换空间。

可使用类似表达，但必须有事实支撑：

- “从风险区回到主动区”。
- “积分还可以，但内容有隐患”。
- “续命成功，但问题没消失”。
- “稳定派”。
- “路径变硬”。
- “结果合格，内容待验证”。

## 11. 发布前自检清单

发布前必须逐项自检：

- [ ] 本期赛果是否全部属于同一个北京时间窗口？

## 12. 发现错误时的处理

如果运行中发现历史日报或刚生成日报存在赛程、日期、xG 或命名错误：

1. 优先修正当前最新日报。
2. 对历史日报，如不宜大规模重写正文，应新增集中核查 / 更正记录，并在 README 或规范文件中挂入口。
3. 更新本规则文件或 `source-policy.md`，防止重复错误。
4. 最终回复必须说明：发现的问题、修正方式、未修正原因、commit、回读验证结果。
